ihwaddbridge:Add a bridge instance to a hardware design.

A bridge is an instance of a generic component which does not appear in any library.
In the current implementation, bridge must have two ports:
    'mp' is the master port,
    'sp' is the slave port.

A bridge is a passive component with no explicit behavioral description. If you
require an active component, see ihwaddmmc.

A bus access by a master on the bus connected to the bridge slave port which
lies in the slave port's address range will be mapped through the bridge
resulting in an access on the bus to which the bridge's master port is
connected.

The address generated on the bus connected to the master port is:
     <address on slave bus> - <slave port loaddress> + <master port loaddress>.

NB - An address range must be supplied for the master port as well as for the
slave port. The two address ranges must be of equal size. Both buses must have
sufficient address bits to represent the hiaddress values.

Arguments:
-help 
    Print this help message
-instancename  <string> (mandatory)
    The name for the new bridge instance
Example Files:

  • Examples/PlatformConstruction/moduleHierarchyAndConnectivity/topLevel/module.op.tcl

ihwaddbus:Add a bus instance to a hardware design.

A bus is an instance of a generic component which does not appear in any
library.
A bus is an abstraction of the wires, decoders and arbitration logic required to
implement the bus and its protocols.
A bus is a passive component; it does not model concurrency, arbitration or
timing.
If you require an active bus model see ihwaddmmc.
The -imagefile option can be used to load data into memory in the bus's address
space.
This option is typically used if the data to be loaded is not associated with
one particular processor.
To load a program associated with a processor, see ihwaddprocessor.

Arguments:
-addresswidth  <integer expression> (mandatory)
    Width of the address in bits
-help 
    Print this help message
-imagefile  <string>
    The image file to load into the address space defined by the bus

ihwaddclparg:Add an argument to the command line parser.

Add an argument to be accepted by the parser. -type must be one of:
      boolvar    : Argument sets variable of type Boolean
      int32var   : Argument sets variable of type Int32
      uns64var   : Argument sets variable of type Uns64
      doublevar  : Argument sets variable of type Double
      stringvar  : Argument sets variable of type String
      boolfunc   : Function called each time flag is specified
      int32func  : Function called each time Int32 argument is specified
      uns64func  : Function called each time Uns64 argument is specified
      doublefunc : Function called each time double argument is specified
      stringfunc : Function called each time string argument is specified
      argvfunc   : Function called with each list of arguments
      stringlist : One list element for each use of the argument
      stringglob : One list element for each use of the argument. Strings are file-glob expanded

Arguments:
-defaultvalue  <string>
    Optional default value for the argument.
-description  <string> (mandatory)
    A description to appear with -help
-group  <string>
    Optional group for the argument (groups clarify the -help output)
-help 
    Print this help message
-mustbespecified 
    If specified, the argument must be specified on the command line.
-name  <string> (mandatory)
    The argument name (do not include the leading dash '-')
-shortname  <string>
    Optional single-character name
-type  <string> (mandatory)
    The type of argument. See above.
-userdata  <integer>
    Pass this value to the callback function.
-valuedescription  <string>
    An optional one word description of the value that follows the argument
Example:
ihwaddclparg -name myarg -description {Example of a boolean} -type boolvar

ihwaddfifo:Add a FIFO instance to a hardware design.

A FIFO is a generic component which does not appear in any library.
It is a point-to-point connection between two processors.
FIFOs can only be used between processors which have FIFO ports and have
FIFO instructions.
/The width and depth of a FIFO is determined by the connected models.
A FIFO operation will always transfer exactly one word into or out of the FIFO.
Behaviour of a FIFO when full or empty is determined by the connected models.

Arguments:

-instancename  <string> (mandatory)
    The name for the new FIFO instance
-width  <integer expression>
    The width in bits of one FIFO word (Currently not used. Width is determined by the connected models)

ihwaddimagefile:Add a new image file to be loaded onto a bus or processor instance.

ihwaddimagefile adds one extra file to be loaded onto a processor instance or bus.
If a file is loaded onto a processor, it is in fact loaded into the memory which is mapped
onto the appropriate address in the code domain of that processor.
If a file is loaded onto a bus, it is loaded into memory which is mapped
onto the appropriate address in the bus domain.
If more than one file is loaded onto a processor, the processor takes its start-address
from the first one loaded.

Arguments:

    The name of the imagefile tag

    Set the PC to the entry point in this object

ihwaddmemory:Add a memory instance to a hardware design.

Add to a design an instance of a memory.

The memory has a single slave port 'sp1' that must be connected to a bus.
The size of the address range (low address to high address) on the slave port
determines the memory's size.
If -privilege is not specified, memory of type ROM will allow read and execute access,
memory of type RAM will allow read, write and execute access.
Memory components are passive; they have no behavioral code. Memory space is
not reserved in the host computer until memory is written.
The simulator contains generic models of RAM and ROM, which can be used for most passive memory modelling. If you require an active model, see ihwaddmmc.

Arguments:
-help 
    Print this help message
-instancename  <string> (mandatory)
    The name for the new memory instance
-library  <string>
    The VLNV library name of the memory
-privilege  <string>
    'n' or any combination of 'r' (read) 'w' (write) and 'x' (execute). e.g. -privilege rx. 'n' creates memory with no access.
-type  <string> (mandatory)
    The VLNV name of the memory type; 'RAM', 'ROM' or 'TRAP'
-vendor  <string>
    The VLNV vendor name of the memory
-version  <string>
    The VLNV version number of the memory

ihwaddprocessor:Add a processor instance to a hardware design.

Add to a design, an instance of a processor from a library.

Specify either full VLNV or modelfile.

The 'semihost*' arguments are used to specify a library of functions which
modify the processor's behavior to provide system I/O without hardware support.
e.g. Code running on the processor which calls 'write' can result in output to
stdout without the existence in the simulator of a UART or other I/O device.

On startup a processor starts at one of three addresses (in order of priority):
1. The address specified with the -startaddress argument.
2. A start address, if specified, in the image file supplied using the
   -imagefile option.
3. The hardware start or reset vector as specified in the model.

The -id argument assigns a unique machine readable identification number to this
processor instance, but this will only have any affect if the processor model
supports this feature.
It is typically used to let software in a symmetric multi-processor system find
out which processor it is running on.
The id is usually accessed by a special processor instruction or by reading a
special read-only register.

The -simulateexceptions flag is set to force the processor to behave like the
real hardware when an exception (e.g. divide-by-zero) occurs. Without the flag,
the simulation stops and a processor exception message is produced.

-alternategdbflags  <string>
    Use different gdb flags for this processor
-alternategdbpath  <string>
    Use a different gdb for this processor
-debugbasic  <optional boolean expression>
    Do not attempt to use GDB with this processor
-debugbitmask  <integer expression>
    Debug flags
-defaultsemihost  <optional boolean expression>
    This processor should load the default semihost library
-dictionarysize  <integer expression>
    Override the default code dictionary size
-enableintercepts  <optional boolean expression>
    Enable Imperas Intercepts on this processor
-endian  <string>
    Select processor endian (big or little)
-help 
    Print this help message
-id  <integer expression>
    A processor ID that is unique within the hardware design
-imagefile  <string>
    The image file to load onto the processor
-instancename  <string> (mandatory)
    The name for the new processor instance
-library  <string>
    The VLNV library name of the processor
-loadphysical  <optional boolean expression>
    Use physical ELF address
-mips  <real expression>
    The nominal MIPS for the processor
-modelfile  <string>
    Path to the processor model file, used instead of VLNV
-semihostlibrary  <string>

imodelnewmmc:Create a new memory mapped component model.

Create a new memory model component (MMC).

An MMC has behavioral code which can model dynamic address decoding (e.g.
plug-and-play), cache or burst behavior.

An MMC is triggered by a bus access request on one of its slave ports and can if
required originate another bus access on one of its master ports.

Some MMCs (e.g. a cache) can have two models of operation, specified by the
'-transparent' flag (see ihwaddmmc).

In transparent mode, the model records the effect of each bus access (e.g.
cache hit/miss) but the actual bus access goes directly onto the other bus.
This mode is faster, but cannot model cache coherency.

In non-transparent mode, the model must generate the bus access on the other bus
if one is required. This mode is slower but can fully model cache behavior.

Having created the MMC, refer to the component modelling guide to create the
code which implements the MMC's behavior.

A processor model must have at least one bus master port, which must be called
'INSTRUCTION'. The second bus master port must be called 'DATA'.
It can also have net input ports for accepting interrupts, and for reset
behavior.

The processor model also describes its stack behavior and some aspects of the
ABI, so that debuggers can understand the stack frames.

Note, the default semihosting library is only used when no platform is
specified, i.e. when only an image file is supplied and the processor type is
derived from that file.

Having created the processor model, refer to the processor modelling guide to
create the code which implements the instruction set simulator.

Arguments:
-defaultsemihostlibrary  <string>
    The VNLV library name for the default Semihosting library
-defaultsemihostname  <string>
    The VNLV name for the default Semihosting library
-defaultsemihostvendor  <string>
    The VNLV vendor name for the default Semihosting library
-defaultsemihostversion  <string>
    The VNLV version number for the default Semihosting library
-elfcode  <integer> (mandatory)
    The code in the ELF file header which identifies the processor architecture
-endian  <string> (mandatory)
    Specifies that the processor is big-endian little-endian or either
-gdbflags  <string>
    Command line flags to be passed to gdb specified in gdbpath
-gdbpath  <string>
    The name of the gdb executable to be used with this model
-help 
    Print this help message
-imagefile  <string>
    The name of the image file that implements the model
-library  <string>
    The VLNV library name for the new processor model
-loadphysicaladdress 
    Use the physical ELF address
-name  <string> (mandatory)
    The VLNV name for the new processor model
-releasestatus  <string>
    Set the current release status
-useindefaultplatform 
    If there is a choice, use this processor in a default platform
-vendor  <string>
    The VLNV vendor name for the new processor model
-version  <string>
    The VLNV version number for the new processor model
-visibility  <string>
    Set the model visibility
Example:
imodelnewprocessor -elfcode intvalue -endian stringvalue -name stringvalue

imodelnewsemihostlibrary:Create a new semihosting library.

A semihosting library packages an image file whose function is to intercept
op-codes, addresses or symbols in code executed by a simulated processor to
provide access to host computer resources.

Typical semihosting functionality might allow reading of host disk files, and
output to a log file without access to models of a disk or display adapter.

A semihosting library is referred to by VLNV.

Refer to the semihosting modelling guide for more information.

Arguments:
-autoload 
    Do not use this flag
-help 
    Print this help message
-imagefile  <string>
    Optional name of the imagefile file that implements the semihosting
-library  <string>
    The VLNV library name for the semihosting library
-name  <string> (mandatory)
    The VLNV name for the semihosting library
-releasestatus  <string>
    Set the current release status
-userheader  <string>
    Path to header file to include at the top of each output file
-vendor  <string>
    The VLNV vendor name for semihosting library
-version  <string>
    The VLNV version number for semihosting library
-visibility  <string>
    Set the model visibility
Example:
imodelnewsemihostlibrary -name stringvalue
